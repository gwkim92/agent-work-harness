# 리서치 기반 핵심 원칙

이 문서는 로컬 리서치에서 반복적으로 나타난 패턴만 추려서 "프로젝트 이식용 원칙"으로 재구성한 것이다.

## 1. 프롬프트보다 작업장이 중요하다

좋은 결과는 긴 프롬프트보다:

- 읽기 쉬운 루트 맵
- 명확한 경계
- 고정된 검증 루프
- 구조화된 handoff

에서 더 많이 갈린다.

이 원칙은 `하네스_엔지니어링_통합_가이드.md`의 중심 결론이다.

## 2. `AGENTS.md`는 짧아야 한다

루트 파일에는 아래만 남긴다.

- 목적
- 디렉터리 맵
- 핵심 명령
- 경계
- 작업 규칙
- 완료 조건

깊은 설명을 다 집어넣으면 오히려 읽기 비용이 커진다.

이 원칙은 `하네스_엔지니어링_통합_가이드.md`와 `gstack_agent_reference.md`에서 강하게 반복된다.

## 3. 생성과 평가는 분리해야 한다

같은 느슨한 루프가:

- 구현도 하고
- 완료도 선언하고
- 품질도 스스로 승인하면

과신과 누락이 생기기 쉽다.

따라서 최소한 논리적으로라도:

- generator
- evaluator

를 분리해야 한다.

웹이라면 브라우저 검증이 evaluator 역할을, 백엔드라면 테스트/로그/실요청 검증이 evaluator 역할을 한다.

## 4. 추론과 제어를 분리해야 한다

모델은 제한된 범위 안에서 판단하게 두고, 아래는 코드나 문서로 고정한다.

- 단계 순서
- 정지 조건
- pass/fail 기준
- 승인 지점

이 원칙은 `ADK_에이전트_설계_가이드.md`의 핵심이다.

## 5. 수정 가능한 표면을 좁혀야 한다

에이전트가 잘 작동하려면 "무엇을 바꿔도 되는가"를 줄이는 편이 좋다.

좁힐 수 있는 표면:

- 수정 가능한 파일
- 실행 가능한 명령
- 실험 시간
- 비교할 metric

이 원칙은 `karpathy_autoresearch_상세분석_및_활용가이드.md`에서 매우 선명하게 드러난다.

## 6. 검증 하네스는 고정해야 한다

작업마다 완료 기준이 흔들리면 에이전트 품질이 아니라 평가 기준이 흔들리는 것이다.

프로젝트는 최소한 아래 질문에 항상 답할 수 있어야 한다.

- 어떤 명령이 기능이 동작함을 증명하는가
- 어떤 검증이 기존 동작을 깨지 않았음을 증명하는가
- 어떤 런타임 또는 브라우저 확인이 사용자 관점 동작을 증명하는가

## 7. 새 기술 도입은 어댑터 뒤에 숨겨야 한다

새 라이브러리, 새 SDK, 새 프레임워크는 바로 전면 채택하지 않는다.

먼저:

- 내부 인터페이스 뒤에 감싼다
- 작은 파일럿에 붙인다
- 기존 경로와 oracle 비교를 한다
- 통과하면 확대한다

이 원칙은 `pretext-adoption-report.md`와 `agentscope_detailed_report.md` 모두에서 유효하다.

## 8. 긴 작업은 handoff artifact가 필요하다

세션이 끊길 때마다 채팅을 다시 뒤지는 구조는 취약하다.

적어도 아래는 파일에 남겨야 한다.

- 현재 상태
- 건드린 파일
- 결정한 것
- 검증한 것
- 다음 한 단계

## 9. UI 작업은 실제 실행 증거가 우선이다

UI, 상호작용, 브라우저 상태가 중요한 작업은 코드만 보고 완료 선언하면 안 된다.

가능하면:

- 실제 페이지 진입
- 상호작용 경로 확인
- 회귀 시나리오 확인

이 필요하다.

이 원칙은 `gstack_agent_reference.md`에서 브라우저 중심 흐름으로 잘 드러난다.

## 10. 멀티에이전트는 기본값이 아니다

새 프로젝트에 가장 좋은 시작점은 대개:

- 짧은 `AGENTS.md`
- 고정 verification
- contract
- handoff

를 갖춘 guided single-agent다.

planner, multi-agent, automation은 실제 병목이 드러난 뒤 붙이는 편이 안전하다.

## 11. 멀티에이전트는 자유형보다 패턴 카탈로그로 다뤄야 한다

multi-agent가 필요해졌다고 해서 역할을 즉흥적으로 늘리면:

- ownership이 겹치고
- handoff가 사라지고
- topology가 말로만 존재하고
- coordinator가 모든 혼란을 떠안게 된다

따라서 multi-agent는 "역할 수를 늘리는 행위"보다:

- 어떤 패턴을 쓰는가
- 어떤 artifact를 주고받는가
- 누가 최종 verification gate를 쥐는가

를 먼저 고정하는 편이 낫다.

권장 패턴은 별도 문서로 유지한다.

- `guides/multi_agent_patterns.md`
